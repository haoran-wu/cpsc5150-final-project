# Report: Using SMT for Guardrailing LLMs on 26 U.S.C. § 152

**Hardware/software for reproduction**: Tested end-to-end on the CPSC 4150/5150 Yale Bouchet OnDemand GPU environment using 1 NVIDIA RTX 5000 Ada GPU, 1 CPU core, 12-24 GB memory, and Python 3.12. The code is path-independent and can be run from either a SLURM job or a course OnDemand Jupyter terminal. Package versions are pinned in `requirements.txt` (`torch 2.3.1`, `transformers 4.40.2`, `trl 0.8.6`, `peft 0.10.0`, `bitsandbytes 0.43.1`, `z3-solver 4.12.6.0`). Run `python scripts/run_smt.py` for SMT and `bash run_course_session.sh all` for training/evaluation.

I chose **LoRA SFT** rather than DPO. With only 10, 20, and 50 examples, SFT was more stable, faster to debug, and easier to reproduce in the course environment. DPO would use the preference pairs more directly, but it adds a reference-policy objective and was more sensitive to TRL version differences and prompt formatting. LoRA with 4-bit NF4 quantization kept the run within single-GPU memory limits. The tradeoff is that SFT trains only on the `chosen` answers, so it does not directly penalize the `rejected` answers.

The datasets were built manually from 26 U.S.C. § 152, my Project 4 SMT formalization, and hand-written legal analyses. For each example, the `chosen` response states the legally correct result and cites the controlling test; the `rejected` response encodes a realistic mistake such as ignoring the age rule, misreading the gross-income threshold, confusing qualifying child with qualifying relative, or treating an equality boundary as sufficient. The six provided evaluation fact patterns, and close paraphrases of them, were excluded from training.

For SMT evaluation, I used a double-query method on each fact pattern: one solver run asserts `dependent`, and a second asserts `(not dependent)`. This avoids treating a single satisfying model as a definitive legal answer. Using the Project 4 exemption amount of $5,000, the SMT verdicts were: Alex Yes, Maria No, James Yes, Linda No, Carlos No, Sophie Yes. Sophie required a small extension for eligible foster child treatment under §152(f)(1)(C).

Performance was stable across the ablation: small, medium, and large each achieved **3/6 correct**. More data did not consistently improve the final verdict labels, although the larger model followed the requested answer format more reliably on Sophie. The model reliably handled Alex, James, and Sophie, which are the three SMT-Yes cases.

The main weakness is a strong **Yes-bias**. The model often latches onto family support or co-residence and answers “Yes” without enforcing the statute’s numerical gates. It misses Maria’s age limit, Carlos’s income problem, and Linda’s strict less-than rule in every run. The outputs also contain occasional unreliable citations, even when the final Yes/No label is correct. These repeated failures show why the SMT guardrail is useful: it cleanly rejects confident but legally incorrect model outputs and exposes exactly which statutory condition failed.
